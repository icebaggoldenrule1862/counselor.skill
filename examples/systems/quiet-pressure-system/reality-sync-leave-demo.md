# 蒸馏后如何现实同步：请假示例

这份样例不是为了说明“请假是核心题材”，而是为了展示：

> 当用户已经蒸馏出自己的辅导员后，系统如何把排练结果同步到现实里。

## 1. 用户当前困境

用户不是单纯想“模拟一下”。
真实需求是：

- 明天需要请假
- 理由不算特别好说
- 这位辅导员对临时请假比较敏感
- 用户不敢乱发第一条消息

## 2. 这时为什么要先蒸馏

如果没有蒸馏，系统只能给一个泛化版本：

- 也许太软
- 也许太模板
- 也许忽略了这位辅导员对时间线和补材料的敏感点

而蒸馏之后，这位辅导员系统已经知道：

- 她私下不会一上来把话说死
- 但会慢慢收口
- 她很看重时间线完整
- 如果理由不清，会往“补材料 / 写说明”推进

## 3. 系统此时最该给什么

不是长篇分析，而是：

1. 一条能发的第一条消息
2. 如果被追问，最稳的下一句
3. 一句提醒：她最可能卡你哪

## 4. 一个更像现实同步的输出

### 第一条消息

老师，我想先跟您说一下，明天上午那节课我这边有个临时情况，可能需要请假处理。我知道这种事不适合临时说，所以先提前跟您报备一下，具体情况我可以马上补充说明。

### 如果对方回“具体说”

老师，是这样的，我这边现在已经在处理了，不是想临时糊弄过去。我把具体时间线和需要补的材料整理一下，马上一起发给您。

### 为什么这更像蒸馏后的结果

因为它同时体现了这位辅导员的几条真实逻辑：

- 第一轮不会被无故解释拖很长
- 她对“为什么现在才说”很敏感
- 她会很在意时间线和补材料
- 所以系统不会教用户一上来就情绪化长篇解释

## 5. 拿去现实里用后要带回什么

如果用户真的发了，回来时最重要的不是说“成没成功”，而是带回：

- 导员真实怎么回的
- 她第一句卡的是什么
- 她有没有立刻要求补材料
- 她到底是给了台阶，还是已经准备留痕

这些会继续进入：

- correction
- session
- archive

## 6. 这个示例想证明什么

这个仓库真正想做的不是“写个请假模板”。
它想证明的是：

> 当蒸馏足够真时，系统可以把一个现实里不敢直接开的口，先在这里练到更稳，再更自然地同步回现实沟通。
