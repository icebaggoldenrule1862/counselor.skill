# 同一事件对比：请假，在不同系统模式下会怎么完全不同

这份样例专门证明一件事：

> 这个项目不是几个固定模板，而是同一个事件在不同“系统模式”下会变成完全不同的工作面。

同样都是“请假”，用户真实遇到的困境可能完全不同。

## 模式 1：开口前排练

### 用户状态

- 还没发消息
- 不知道第一句怎么说
- 最怕一开口就显得太虚

### 系统应该输出什么

- 3 个开场版本
- 每个版本的风险点评
- 导员最可能的第一轮追问

### 输出样例

```text
老师好，我想跟您请个假，时间是明天下午。
事情比较临时，我先把具体情况和会影响到的课程跟您说清楚，您看看这样处理是否合适。
```

### 重点

这里的核心不是“请假理由合不合理”，而是：

- 第一条消息怎么不失控
- 口气怎么像“申请”而不是“通知”

## 模式 2：被追问后的应对

### 用户状态

- 已经发出去了
- 导员回了“具体说”
- 用户不会接下一句

### 系统应该输出什么

- 当前这一轮可直接回的消息
- 再下一轮的预判
- 哪些说法会继续降信任

### 输出样例

```text
老师，是这样，我明天下午这边临时有个必须处理的事，所以想请半天假。
我知道这次说得比较临时，所以先跟您把时间和课程影响说明白，如果需要补材料或者跟任课老师同步，我这边也配合。
```

### 重点

这里已经不是“开口”问题，而是：

- 怎么把模糊理由说具体一点
- 怎么把态度拉稳

## 模式 3：聊崩后的补救

### 用户状态

- 前面已经说得不太好
- 导员明显有点不爽
- 用户怕事情升级

### 系统应该输出什么

- 一条补救消息
- 说明现在该先处理态度还是先处理事实

### 输出样例

```text
老师，刚刚那条是我说得不合适，我重新说明一下。
我不是想把这个事情随便带过去，是我刚刚一下子有点慌，说话没整理好。
我把具体情况重新跟您说清楚，您再看怎么处理比较合适。
```

### 重点

这里不是“请假怎么说”，而是：

- 态度崩了以后怎么先把局面拉回来

## 模式 4：办公室高压预演

### 用户状态

- 明天要去办公室
- 导员手里可能已经有任课老师反馈
- 用户想先过一遍最可能的问答

### 系统应该输出什么

- 3 到 5 轮高压预演
- 证据分步揭示
- 风险上升路径

### 输出样例

```text
周老师：
你请假的事我看到了。
我现在问你的不是“有没有事”，而是为什么这件事你是在临近上课才说。

学生：
老师，确实是临时情况。

周老师：
临时情况可以理解，但临时沟通和临时通知不是一回事。
你先把时间线说清楚，我再判断你这个情况该怎么记。
```

### 重点

这里已经进入“高压场景”：

- 不是帮你润一句话
- 而是在模拟整个压力结构

## 模式 5：说明文工坊

### 用户状态

- 沟通已经发生过
- 现在需要补一份说明

### 系统应该输出什么

- 一份像学生真实会写的说明
- 一份更正式的版本

### 输出样例

```text
老师您好，关于我 4 月 8 日下午临时请假的情况，补充说明如下：
当天下午家里临时有需要我立即处理的事情，我在临近上课前才意识到时间上会冲突，因此沟通得比较仓促。
这次处理上确实不够规范，后续如需补充材料或同步任课老师，我会继续配合。
```

### 重点

这里的任务已经不是对话，而是：

- 把前面的沟通结果沉淀成可提交文本

## 结论

同样是“请假”：

- 开口前排练：解决的是“怎么开口”
- 被追问后的应对：解决的是“下一句怎么接”
- 聊崩后的补救：解决的是“怎么救回来”
- 办公室预演：解决的是“明天会怎么谈”
- 说明文工坊：解决的是“后续材料怎么写”

这就是为什么这个项目不能只按事件分。
真正的核心是：

> 同一个事件，在不同系统模式下，对用户来说根本不是同一个任务。
