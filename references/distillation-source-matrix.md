# 蒸馏来源矩阵

这份文档定义的是：

> 辅导员系统可以通过哪些不同来源被蒸馏，以及每类来源最适合提取什么？

这部分明显参考了 `ex-skill` 的多源输入思路：

- 聊天记录
- 社交文本
- 照片与时间线
- 用户手动描述
- 后续 correction

但在辅导员场景里，我们要把它翻成更适合高校环境的来源矩阵。

## 核心原则

不要把“蒸馏”理解成只读一份通知。

成熟版本里，蒸馏应该至少分成这几层：

1. 规则层
2. 话术层
3. 施压层
4. 协调链路层
5. 用户主观印象层

不同来源会贡献不同层。

## 来源 1：通知 / 班群公告

典型形式：

- 班群通知
- 学院提醒
- 考勤提醒
- 宿舍管理通知

最适合提取：

- 官话浓度
- 高频词
- 命令句倾向
- 制度边界
- 常见后果表达

适合进入：

- `speech_style`
- `rule_universe`

## 来源 2：私聊聊天记录

典型形式：

- 辅导员私聊
- 追问记录
- 约谈前后短消息

最适合提取：

- 追问方式
- 常用开场
- 亮证据节奏
- 是先压态度还是先问事实

适合进入：

- `pressure_style`
- `speech_style`
- `coordination_style`

## 来源 3：班会发言 / 办公室谈话转写

典型形式：

- 班会讲话稿
- 约谈记录
- 语音转文字
- 办公室聊天纪要

最适合提取：

- 长段表达习惯
- 施压节奏
- 升级路径
- 喜欢先讲集体影响还是个人问题

适合进入：

- `pressure_style`
- `escalation_style`
- `mode_effects.office-talk`

## 来源 4：制度文本 / 学院规定

典型形式：

- 学生手册
- 请假制度
- 考勤规则
- 宿舍管理规定

最适合提取：

- 规则硬度
- 书面说明/约谈/上报的升级路径
- 哪些红线会直接拉高风险

适合进入：

- `rule_universe`
- `scene_redlines`
- `escalation_watchpoints`

## 来源 5：说明文批注 / 审批反馈

典型形式：

- “重写”“不完整”“先补材料”
- 请假审批备注
- 导员对情况说明的文字反馈

最适合提取：

- 导员最在意的材料缺口
- 对书面说明的标准
- 更重态度、事实还是流程

适合进入：

- `statement_style`
- `follow_up_requirements`

## 来源 6：用户手打印象 / 口述回忆

典型形式：

- “我们导员表面温和，但其实特别会慢慢收口”
- “她不会一上来凶人”
- “他很爱拿任课老师反馈压人”

最适合提取：

- 典型行为规则
- 口头禅
- 用户高置信度 correction 候选

适合进入：

- `manual_profile`
- `correction_candidates`

## 来源 7：案例材料 / 处分流转痕迹

典型形式：

- 某次违纪处理通知
- 某次约谈结果
- 某次班委反馈整理

最适合提取：

- 真正的处理路径
- 哪类事情会进入正式记录
- 升级是否快、是否给台阶

适合进入：

- `case_patterns`
- `escalation_style`

## 来源 8：多轮 correction 本身

这也是一种蒸馏来源。

因为长期 correction 本身就说明：

- 用户心里的导员到底长什么样
- 哪些行为必须保留
- 哪些行为必须禁止

适合进入：

- `stable_constraints`
- `mode_effects`

## 建议的来源权重

推荐默认权重：

1. 用户 correction
2. 用户手打印象 / 口述
3. 私聊聊天记录 / 办公室谈话转写
4. 通知 / 规定 / 批注
5. 其他辅助来源

原因很简单：

- 规定能说明规则
- 通知能说明口气
- 但“这个导员究竟像不像”，用户本人通常最有判断力

## 最终目标

成熟版本里的蒸馏，不是“把一份文本扔进去”。

而应该更像：

> 多种来源分别提取不同层信号，再统一 merge 成一个更像真实学校生态的辅导员系统。
