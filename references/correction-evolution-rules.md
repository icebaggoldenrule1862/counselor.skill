# Correction 进化规则

这份文档定义的是：

> 一次 correction，应该怎样稳定影响整个辅导员系统，而不是只改一轮输出？

如果 correction 只能修一次对话，它就只是临时补丁。
真正成熟的版本里，correction 应该更像系统进化规则。

## 1. correction 的优先级

同一套系统里，建议按下面顺序决定行为：

1. 用户手动 correction
2. 用户手动人格设定
3. 蒸馏材料中稳定出现的证据
4. archetype 默认值

这意味着：

- 用户说“他不会这样说”时，优先相信用户
- 但如果多份蒸馏材料稳定证明某个行为，也不能完全忽略

## 2. correction 优先改行为，不优先改标签

优先修改：

- 第一轮怎么开口
- 追问时怎么压
- 什么时候亮证据
- 什么时候提班长、任课老师、家长
- 什么时候给台阶

不要优先修改成：

- 太凶
- 太软
- 太官

因为这些词如果不落到行为，就很难跨模式生效。

## 3. correction 必须跨模式同步

一个成熟的 correction，不应只影响“办公室预演”。

它至少应同步到：

- 开口器
- 追问应对器
- 办公室推演器
- 复盘器

例如：

- 如果 correction 说“不会一上来把话说死”
  - 开口器：第一轮追问更像试探
  - 追问应对器：先听时间线，再收口
  - 办公室推演器：证据分步揭示
  - 复盘器：更强调“在哪一轮开始变硬”

## 4. correction 的稳定性判断

建议把 correction 分成两类：

### 强 correction

满足任一条件：

- 用户明确说“他绝不会这样说”
- 用户提供了稳定反复出现的典型行为
- correction 针对的是高频核心模式

强 correction 应直接覆盖旧行为。

### 弱 correction

例如：

- “感觉更像稍微温和一点”
- “可能不会这么快提家长”

弱 correction 更适合做为偏好微调，而不是完全覆盖。

## 5. correction 的影响范围

建议至少记录这几层：

- `speech_style`
- `pressure_style`
- `evidence_reveal`
- `coordination_style`
- `mode_effects`

其中最重要的是 `mode_effects`，因为它直接描述：

- correction 会怎样改写开口器
- 怎样改写追问应对器
- 怎样改写办公室推演器
- 怎样改写复盘器

## 6. correction 不应破坏系统连续性

修正时不要把旧系统完全推翻。

正确做法是：

- 保留原 archetype 的大方向
- 在关键行为节点上修正
- 让修正后的系统“更像”，而不是“变成另一个人”

## 7. correction 应进入展示层

成熟版本里，correction 不该只藏在 persona.md 里。

它还应进入：

- `snapshot.json`
- 长期档案的观察笔记
- 前后对比样例

这样仓库访客才能看出：

- 这是一个会进化的系统
- 不是一组静态 prompt

## 最终目标

这层规则真正想做到的是：

> 一次 correction 之后，整个系统在多种模式下都会更像用户心里的那个辅导员，而不是只改一局台词。
