# 系统生命周期

这份文档回答的不是“有哪些模式”，而是：

> 一个专属辅导员系统从第一次创建到长期使用，究竟会经历哪些阶段？

如果没有这层生命周期设计，项目很容易退化成：

- 几个独立 prompt
- 一些孤立场景
- 一堆看起来很多、但互相不接的功能块

这个项目真正要做成的，是一个会持续长、持续被修、持续被使用的系统。

## 1. 启动阶段：先让用户立刻进场

目标：

- 不要求用户先理解整个系统
- 不要求用户先上传大量素材
- 不要求用户先挑一堆事件分类

入口应该优先围绕用户当前困境：

- 我还没开口
- 我已经发出去了但不会接
- 我明天要被约谈
- 我刚刚聊崩了
- 我需要写说明文
- 我其实是要求助

这一阶段的交付标准是：

- 2 分钟内看到第一个可用输出
- 先解决眼前沟通问题，再决定是否深入定制

## 2. 建模阶段：生成“这个辅导员系统”

目标：

- 给用户一个可控的辅导员系统
- 而不是只给一次临时回复

建模来源按优先级分层：

1. 用户手动设定
2. 用户 correction
3. 蒸馏材料
4. archetype 默认值

这一阶段要形成至少两类核心文件：

- `persona.md`
- `meta.json`

理想状态下，用户第一次就能感到“像某种真实导员”，而不是抽象标签集合。

## 3. 路由阶段：根据当下困境切模式

系统不应该先问“你要模拟哪个事件”，而应该先判断：

- 现在最需要哪种工作模式
- 是否要同时调用多个模式

例如：

- “明天要被约谈，今晚想先准备”
  路由到：办公室推演器 + 复盘器
- “刚给导员发了消息，对面回了‘具体说’”
  路由到：追问应对器
- “不是违规，是最近状态不太好”
  路由到：求助沟通器 + 开口器

这一步是项目避免模板化的关键。

## 4. 运行阶段：完成一次具体互动

运行阶段的目标不是把话说得多，而是把系统工作做对。

一次合格的运行，至少要完成：

- 明确当前沟通节点
- 明确对方可能在意什么
- 明确当前风险
- 给出可直接用的输出
- 在需要时模拟 2 到 5 轮推进

如果进入深场景，还应带上：

- 案件状态
- 证据揭示节奏
- 风险升级路径
- 结果档位判断

## 5. 沉淀阶段：把本轮变成记忆

一次使用结束后，不应该像没发生过一样。

系统至少要能沉淀：

- 这次练了什么
- 用户卡在哪
- 哪种说法更适合用户
- 哪个模式更有效
- 哪些表达明显会触发风险

这一步的核心产物是：

- `sessions/{timestamp}.md`

如果没有这层，系统很难真正产生长期价值。

## 6. 纠偏阶段：用户可以改系统

用户说“不像”“太凶了”“我们导员不会这么说”时，不应该只解释原因。

正确动作是：

- 把反馈写成 correction
- 优先改行为风格，而不是空泛标签
- 立刻给出修改后的短预览

核心产物：

- `corrections/{timestamp}.json`

这一层决定了项目是不是“可控系统”。

## 7. 进化阶段：系统不只是记住，还会变化

长期来看，系统应该表现出：

- 用户练得越多，输出越贴近用户风格
- correction 越多，导员人格越稳定
- 蒸馏材料越多，制度语气和证据逻辑越像某个真实学校环境

这意味着系统的变化不应该只体现在一次回复里，而应该体现在：

- 后续模式切换
- 后续场景推进
- 后续复盘重点
- 后续压力呈现方式

## 8. 展示阶段：让 GitHub 用户一眼看懂

这个项目既然要发布到 GitHub，上层展示也属于系统生命周期的一部分。

仓库至少要让访客快速看到：

- 这是一个“系统”，不是几个 prompt
- 系统里已经积累了什么
- 一次使用后会留下什么
- 两三次使用后会怎么变化

推荐展示材料：

- 一个完整系统示例目录
- 一份系统演化 walkthrough
- 一份 snapshot
- 一组同题不同模式对比

## 9. 最终闭环

理想闭环应该是：

1. 用户从当下困境进入
2. 系统路由到合适模式
3. 输出可直接使用的内容
4. 把本轮沉淀成 session
5. 用户随时纠偏
6. 系统下一次明显更像、更稳、更连续

只有走到这一步，这个项目才真正配叫：

> 一个专属辅导员交互系统生成器，而不是一个场景 prompt 集。
