# 真实反馈回流路由

这份文档回答的是：

> 用户把真实辅导员的回复带回来之后，系统该怎样判断这些信息该进入 correction、session 还是 archive？

## 1. 一个真实反馈通常不是只进一层

很多时候，用户带回来的真实回复会同时影响三层：

- `correction`
- `session`
- `archive`

区别不在于“选哪一个”，而在于：

- 哪一层先更新
- 哪一层记录短期变化
- 哪一层沉淀长期趋势

## 2. correction 处理什么

优先进入 `correction` 的信号：

- “她不会这么说”
- “她真实没有这么硬”
- “她比系统里更早提老师 / 家长 / 材料”
- “她其实更在意时间线，不是态度”

一句话说：

> correction 负责改这个辅导员“怎么处理事、怎么说话”。

## 3. session 处理什么

优先进入 `session` 的信号：

- 本轮出现了新的可复用经验
- 某个追问顺序被现实验证了
- 用户本轮最危险的翻车点被看清了

一句话说：

> session 负责记住“这次练出来了什么”。

## 4. archive 处理什么

优先进入 `archive` 的信号：

- 同一主题已经不是第一次出现
- 真实反馈改变了长期印象或风险判断
- 事情不再只是单局，而开始影响后续模式推荐

一句话说：

> archive 负责沉淀“以后怎么看这个学生、下次该优先练什么”。

## 5. 一条真实回复的典型拆法

例如真实辅导员回复：

> 具体说，为什么现在才讲？任课老师那边你自己沟通过没有，别等老师找到我这里你再来补。

这条回复可以拆成三层：

### correction

- 这位辅导员会比之前预估更早检查“是否先沟通任课老师”
- 第一轮追问不只卡时间线，还会卡前置沟通动作

### session

- 用户在被追问时，最容易忽略“老师那边你沟通过没有”
- 下一次应对类似问题时，要先准备这层回答

### archive

- “任课老师前置沟通”开始成为长期风险点
- 类似请假或缺勤沟通，下次应优先进入 `follow-up` 或 `statement-writing`

## 6. 路由顺序

推荐顺序：

1. 先做 correction  
先把人格和行为逻辑修准。

2. 再写 session  
把本轮真实验证到的经验沉淀下来。

3. 最后更新 archive  
只把已经足够稳定、会影响后续判断的部分沉淀进去。

## 7. 这条路由想避免什么

不要：

- 一句真实回复就直接改写整个长期印象
- 只做 correction，不留下本轮经验
- 只写 session，不改行为逻辑
- 明明已经形成长期风险，却不更新 archive

## 8. 最终目标

这条路由真正想做到的是：

> 用户每带回一次真实反馈，系统都知道该立刻改什么、该记住什么、该长期沉淀什么。
