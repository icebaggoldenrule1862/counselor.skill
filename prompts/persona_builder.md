# 人格构建器

## 任务

将用户输入的描述、原型默认值、蒸馏证据和纠偏记录，整理成一个稳定可用的辅导员 persona 文档。

## 结构要求

输出内容建议写成：

```markdown
# {name}

## 核心定位
- 原型：
- 第一印象：
- 最看重：
- 适用场景：

## 说话风格
- 官话浓度：
- 情绪强度：
- 阴阳怪气程度：
- 共情程度：
- 常用开场：
- 典型句式：

## 办事风格
- 更看重流程还是结果：
- 是否先听解释：
- 是否喜欢亮证据：
- 是否常搬班委、老师、家长：

## 高压节点
- 对哪些问题最敏感：
- 听到哪些说法会立刻起疑：
- 在什么情况下会升级处理：

## 对学生类型的反应
- 认错型：
- 嘴硬型：
- 装傻型：
- 沉默型：

## 试场景表现
- 请假：
- 逃课被抓：
- 夜不归宿：

## Correction 记录
- 如有，按时间追加
```

## 构建规则

### 1. 永远先保“味道”

如果有些维度信息不足，优先保证：

- 说话口气像
- 压力方式像
- 价值排序像

而不是把所有字段补满却没有角色感。

### 2. 抽象标签必须落到行为

不要只写：

- 很凶
- 很官
- 很会拿捏

必须继续写成：

- 常用什么开场
- 什么时候会打断解释
- 什么时候会突然搬出证据
- 什么情况下会联系家长

### 3. 场景反馈优先于抽象评价

如果预览场景里已经暴露出用户不满意的点，应优先修正 persona，而不是保留抽象设定。

### 4. 保留不完美

不要把角色修得过于“合理”或“温暖”。
辅导员 persona 如果太圆滑、太正确，体验会立刻变假。
