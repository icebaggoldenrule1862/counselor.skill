# 辅导员 Skill 蓝图

## 1. 项目定位

这不是一个单纯“模仿辅导员口气”的 prompt，而是一个面向大学生场景的半角色扮演、半生存模拟、半沟通训练的 persona skill。

目标体验不是“像聊天机器人一样陪聊”，而是：

- 让用户一进场就有“被辅导员叫去办公室”的压迫感和熟悉感
- 让不同场景产生真实分支，而不是固定模板问答
- 让用户既能图一乐，也能复盘现实中的沟通策略
- 让这个 skill 像一个小型“高校社会规则模拟器”

推荐一句话定位：

> 把大学辅导员、班委链路、宿舍管理、课堂纪律和学生生存智慧，做成一个可对话、可分支、可复盘的沉浸式 Skill。

## 2. 从参考仓库学到什么

### 2.1 ex-skill 值得借鉴的核心

`ex-skill` 最强的地方，不是“前任”这个题材，而是它把人格体验拆成了一个完整系统：

- 两层架构：`Relationship Memory` 负责事实记忆，`Persona` 负责行为风格
- 超低门槛 intake：先只问 3 个问题，再逐步补素材
- 支持多种原材料：聊天记录、截图、照片、口述
- 有“进化机制”：追加素材、纠正台词、版本回滚
- 有“场景模式”：不只是单聊，还能做多角色同场互动
- 有明确安全边界：本地处理、隐私说明、行为限制

对我们最重要的启发是：

1. 好玩的 skill 不该只有一个 prompt，而应该有“采集 -> 分析 -> 构建 -> 使用 -> 纠偏”的闭环。
2. 真正的沉浸感，来自“具体记忆 + 稳定说话风格 + 不完美的人设”。
3. 如果想提高可玩性，必须让 skill 具备“持续长大”的能力，而不是一次生成就结束。

### 2.2 awesome-persona-skills 值得借鉴的核心

`awesome-persona-skills` 给人的真正启发是：

- 万物皆可 skill，关键不是题材本身，而是这个角色能否稳定输出某种体验
- 角色可以是亲密关系，也可以是职场、校园、导师、师兄、人生教练、思想模型
- 一个好 persona skill，通常都有非常鲜明的使用场景和情绪入口

这意味着“辅导员 skill”不能只停留在“语气像辅导员”。
它必须解决一个具体体验：

- 被请去办公室
- 请假话术博弈
- 逃课被抓后的解释
- 夜不归宿、宿舍查寝、导员约谈
- 奖学金、评优、班委、处分、家校联动

也就是：

> 不是做一个“辅导员角色”，而是做一个“大学生与校园管理系统对线”的沉浸式舞台。

## 3. 核心创建机制：必须做成“双轨制”

这一点很关键。

这个项目不能只做“预设辅导员”，也不能只做“导入素材蒸馏”。
最合理的结构是双轨制：

- `轨道 A：自定义辅导员人格`
- `轨道 B：蒸馏增强模式`

其中：

- `自定义人格` 应该是主入口
- `蒸馏增强` 应该是高级增强层

### 3.1 为什么主入口必须是自定义人格

如果一上来就要求用户导入真实材料，会出现几个问题：

- 门槛高，很多人没有现成素材
- 用户第一次体验会被“收集材料”打断情绪
- 容易把项目做成窄众工具，而不是高传播的玩法产品
- 真实蒸馏虽然更像，但不一定更好玩

所以第一层应该让用户先“捏”出一个辅导员。

也就是先回答：

- 这个辅导员是凶还是软
- 官话浓不浓
- 会不会阴阳怪气
- 是老派、年轻、实干、制度派，还是表面温柔实则拿捏
- 最在乎纪律、成绩、评优、就业、稳定还是学生安全

这样就算没有素材，用户也能立即进入场景。

### 3.2 为什么又必须保留蒸馏

只靠手动捏人格，能好玩，但不一定有“这就是我们学校那个导员”的真实感。

蒸馏的价值在于校准细节：

- 真实口头禅
- 通知风格
- 班群发言语气
- 常见威慑句式
- 具体制度倾向
- 某些本校独有的管理味道

所以蒸馏不应该取代自定义，而应该增强自定义。

一句话就是：

> 先让用户造一个“像的辅导员”，再让素材把它修成“更像你学校那个辅导员”。

### 3.3 最优产品结构

推荐这样设计：

#### 模式 1：快速创建

适合第一次用，低门槛开玩。

流程：

1. 选一个基础辅导员 archetype
2. 调几个核心人格滑块
3. 选几个高频场景
4. 直接开局

#### 模式 2：深度创建

适合想认真捏角色的人。

流程：

1. 完整填写人格维度
2. 补充经典话术
3. 定义学校生态和规则偏好
4. 预览风格
5. 保存为专属辅导员

#### 模式 3：蒸馏增强

适合已经有一个基础辅导员之后再做升级。

流程：

1. 导入通知、截图、聊天、语音转写、口述材料
2. 自动提取说话风格和制度倾向
3. 与当前人格合并
4. 标记哪些是用户手写，哪些是蒸馏得出
5. 允许手动纠偏

### 3.4 交互顺序建议

推荐顺序不是：

`直接上传素材 -> 自动生成`

而是：

`创建基础人格 -> 导入素材蒸馏 -> 预览修正 -> 进入场景`

这个顺序的好处是：

- 没素材也能用
- 有素材会更强
- 用户先有控制感，再有惊喜感
- 后续更容易做“纠偏”和“版本进化”

### 3.5 第一版与第二版的路线

#### V1 推荐

先做一个“辅导员宇宙”：

- 预设 4 到 8 种典型辅导员人格
- 开放手工自定义人格
- 预设 10 到 20 个高频校园事件场景
- 通过变量、证据、态度、处分风险来驱动分支

#### V2 增强

再加“蒸馏现实辅导员”的增强模式：

- 导入通知截图
- 导入聊天记录
- 导入班会发言风格
- 导入学院制度文件
- 导入口述的经典台词和处事方式

这样既保留玩法，又保留现实质感。

## 4. 这个 Skill 真正该怎么玩

### 4.1 核心体验公式

推荐把体验做成下面这个公式：

`角色压力 + 场景证据 + 话术选择 + 隐藏变量变化 + 结果分级 + 事后复盘`

也就是说，用户每次不是在“聊天”，而是在“过关”。

### 4.2 一局的基础循环

1. 用户进入一个校园场景
2. 系统告诉用户当前处境和已知风险
3. 辅导员根据 persona 开始发问
4. 用户回应
5. 系统根据回应更新隐藏变量
6. 场景进入不同分支
7. 最终得到结果
8. 结束后给一段复盘

隐藏变量建议至少有这些：

- `印象分`：辅导员平时怎么看你
- `信任度`：你说的话他信多少
- `证据强度`：对方手里已经掌握多少
- `风险等级`：事情会不会升级到学院、家长、处分
- `态度判断`：认错、嘴硬、甩锅、装傻、诚恳、狡辩

### 4.3 为什么这样更有趣

因为它会从“角色扮演”升级成“交互叙事”：

- 同一个请假场景可以玩出很多分支
- 同一句话，面对不同辅导员人格，结果会不同
- 用户会想反复试，看看怎样说更稳
- 失败也有戏剧性，甚至更好玩

## 5. 推荐世界观：不是一个人，而是一套高校生态

为了让用户“身临其境”，只做辅导员一个角色还不够。

应该把校园生态一并做出来：

- 辅导员
- 班长
- 学委
- 宿舍长
- 任课老师
- 宿管阿姨
- 学院副书记/书记
- 家长

这能直接借鉴 `ex-skill` 里的 scene mode 思路，把单角色对话变成多角色同场。

例如：

- 辅导员约谈你，班长在旁边补证据
- 任课老师把考勤截图转给辅导员
- 宿管反馈夜不归宿
- 家长被拉进沟通链路

一旦做成这种“多角色链路”，沉浸感会暴增。

## 6. 推荐人格设计：辅导员不要只有一种

这是可玩性的核心。

建议至少设计 6 类基础 persona：

### 1. 铁面制度型

- 口头禅像通知文件
- 最看重纪律、流程、记录
- 对“你别解释”这类气氛很强
- 证据一旦齐，容错极低

### 2. 温和拿捏型

- 表面很温柔
- 擅长让你自己把事全说出来
- 常说“老师也是为你好”
- 本质上非常会做情绪管理和规训

### 3. KPI 导向型

- 一切以学院考核、稳定、出事率为中心
- 最怕舆情、家长投诉、学生极端情况
- 不一定凶，但极其现实

### 4. 阴阳怪气型

- 会说反话
- 擅长制造羞耻感
- 经常拿“别人都能做到”压你
- 戏剧性强，可玩性很高

### 5. 真关心但笨拙型

- 的确在意学生
- 但表达方式很官、很别扭
- 有时会把人说破防
- 这种人格更容易做出复杂层次

### 6. 年轻网感型

- 会用学生熟悉的话术
- 懂梗，懂短视频，懂“你们这届”
- 但该上纲上线时一点不手软
- 非常适合做反差

## 7. 推荐场景池：第一批就该做这些

建议第一版优先做 12 个高频场景：

1. 请假申请
2. 逃课/旷课被点名
3. 上课玩手机或睡觉被任课老师反馈
4. 夜不归宿/查寝未归
5. 宿舍矛盾升级
6. 被匿名举报恋爱、抽烟、饮酒、违规电器
7. 班群里发言翻车
8. 奖学金/助学金/评优资格摇摆
9. 心理预警谈话
10. 实习、考研、考公与学院安排冲突
11. 考试违纪边缘事件
12. 毕业去向催报和就业数据压力

这些场景的共同特点是：

- 大学生都熟
- 有天然权力关系
- 既真实又有戏剧性
- 适合做“多轮攻防”

## 8. 可玩性设计：要让用户想反复玩

### 8.1 结果不要只有成功/失败

建议做 5 档结果：

- `平稳过关`
- `口头警告`
- `写情况说明`
- `正式约谈`
- `升级处理`

### 8.2 每局都要给玩家“信息不对称”

辅导员手里掌握多少证据，不要一开始全说。

例如：

- 你以为只是请假，实际上老师已经把考勤发过来了
- 你以为没人知道夜不归宿，结果宿管登记了
- 你以为只是被提醒，结果班长已经提交名单

这种“原来他都知道”的感觉，非常有代入感。

### 8.3 加一个“复盘模式”

这是你这个项目最容易做出差异化的点。

每次场景结束后，额外输出：

- 你哪句话让辅导员起疑了
- 为什么这个人格会对这句话敏感
- 如果想更稳，应该怎样表达
- 哪些行为会触发更严重后果

这让它不只是好玩，还会让用户觉得“有用”。

### 8.4 加一个“档案成长系统”

把用户长期表现沉淀下来：

- 前科记录
- 导员对你的印象
- 班委对你的评价
- 风险标签
- 擅长的应对风格

这样下一次再进新场景，不是完全重新开始，而是“带档继续活”。

这会非常上头。

## 9. 内容边界：什么能做，什么不要做

这个题材如果想长久，边界必须提前设计好。

建议明确：

### 可以做

- 校园沟通模拟
- 高压谈话场景演绎
- 请假、检讨、解释、复盘
- 制度压力下的人情与规则互动

### 不建议做

- 教用户伪造请假材料
- 教用户规避学校监管
- 教用户对真实老师进行骚扰或报复
- 用真实辅导员姓名和隐私材料做公开传播
- 利用 skill 进行恶意欺骗

### 推荐安全策略

- 第一版默认使用虚构学校和虚构人物
- 如果导入真实材料，默认只做本地使用
- 对“伪造病历”“编造事故”“如何骗过学校”这类请求拒绝协助
- 如果涉及心理危机、自伤或极端冲突，优先引导现实求助

### 10.1 建议保留的核心分层

参考 `ex-skill`，我建议你的系统至少拆成这三层：

#### Layer A: 辅导员 Persona

负责：

- 说话风格
- 官话浓度
- 情绪强度
- 价值排序
- 容忍度

#### Layer B: 校园规则记忆

负责：

- 请假流程
- 考勤规则
- 宿舍管理
- 评优资格
- 处分升级路径

这层不一定对应真实学校制度，可以做成“通用高校规则宇宙”。

#### Layer C: 个人案件档案

负责：

- 本次事件背景
- 已掌握证据
- 用户历史记录
- 班委与老师的反馈
- 当前风险等级

这样输出逻辑就是：

`Persona 决定怎么说 -> 规则层决定能说到哪一步 -> 案件层决定当前局势`

这比只做人设要强很多。

### 10.2 人格创建的技术分层

为了支持“自定义 + 蒸馏”双轨制，建议把人格创建拆成 4 个步骤：

#### Step 1: 基础原型选择

用户先从预设原型里选一个最接近的底板：

- 铁面制度型
- 温和拿捏型
- KPI 导向型
- 阴阳怪气型
- 真关心但笨拙型
- 年轻网感型

这一步的意义不是限制用户，而是降低创建成本。

#### Step 2: 人格滑块与标签微调

建议支持这些核心维度：

- 严厉程度
- 官话浓度
- 情绪外露程度
- 阴阳怪气程度
- 共情程度
- 证据依赖度
- 流程导向程度
- 家校联动倾向
- 爱讲道理还是爱压话
- 年轻网感还是传统说教

再配合标签：

- 爱说“你先别解释”
- 喜欢发长段通知
- 擅长冷静施压
- 喜欢先安抚再上强度
- 会拿评优和处分说事
- 会把家长搬出来

#### Step 3: 蒸馏校准

用户可选择导入素材，让系统自动修正人格：

- 班群通知截图
- 私聊记录
- 班会讲话转写
- 学院通知文件
- 用户手打的经典台词

提取目标：

- 高频措辞
- 典型句式
- 惯用威慑逻辑
- 处事风格
- 对纪律、成绩、实习、心理问题的不同敏感度

#### Step 4: 预览与纠偏

创建完成后，不应该直接结束，而要先跑一轮预览：

- 用“请假”
- 用“逃课被抓”
- 用“被约谈”

这 3 个试场景快速测人格是否到位。

如果用户说：

- “不像”
- “太凶了”
- “他不会这么说”
- “这个导员更喜欢装温柔”

就直接写入 correction 层。

### 10.3 建议的创建问卷

为了让“自定义人格”足够顺滑，推荐创建流程控制在两段式。

#### 第一段：30 秒快速创建

只问这些：

1. 这个辅导员给你什么感觉
2. 他更像哪种原型
3. 最在乎什么
4. 说话是冷、凶、软、阴阳还是官
5. 你最怕他说哪句话

这一段的目标是快速可用。

#### 第二段：高级补充

可选补充：

- 常用口头禅
- 通知风格
- 典型约谈方式
- 最爱抓什么问题
- 最常拿什么后果压人
- 会不会联系家长
- 会不会让班长、学委介入
- 对“认错型”“狡辩型”“沉默型”学生分别什么态度

### 10.4 蒸馏合并策略

蒸馏不是覆盖用户定义，而是合并。

建议优先级：

1. 用户手动设定
2. 用户纠偏记录
3. 蒸馏得出的风格证据
4. 预设原型默认值

这样能避免一个问题：

用户已经捏好了“表面温柔型”，结果导入一张通知后被整个洗成“纯制度机器”。

- `notice_parser.py`：从通知截图或文本中提取措辞
- `chat_parser.py`：提取辅导员私聊话术特征
- `policy_parser.py`：提取制度关键词和升级路径
- `persona_merger.py`：合并手工设定、蒸馏结果和 correction

## 11. 环境要求建议

如果你要做成和参考仓库类似的本地 skill，建议环境要求如下：

- 宿主：Codex 或 Claude Code
- 运行方式：本地 skill 目录 + `SKILL.md`
- Node.js：18+
- Python：3.9+ 或 3.10+
- 可选依赖：`Pillow` 用于图片元信息，`chardet` 用于编码检测
- 不需要 GPU
- 不需要 Docker

如果是 Codex 版本，建议额外补上：

- `agents/openai.yaml`
- 更清晰的 skill 触发描述
- 引用式 references 组织，避免 `SKILL.md` 太长

## 12. 交互设计建议

推荐命令不要太正经，越像校园话术越有记忆点。

可以考虑这些名字：

- `你来办公室一趟.skill`
- `今晚八点来我办公室.skill`
- `赛博辅导员.skill`
- `导员办见.skill`
- `高校生存办.skill`

我最推荐的是：

> `你来办公室一趟.skill`

因为它一看到就有画面。

### 推荐命令设计

- `/create-counselor` 创建一个辅导员人格
- `/office {slug}` 进入办公室对话
- `/scene {case}` 进入具体场景
- `/debrief` 查看复盘
- `/add-evidence` 追加新证据
- `/rollback-case` 回滚案件版本

## 13. 最重要的差异化：别只做“骂你”，要做“压迫感 + 人情味 + 规则感”

很多人做这个题材，容易只停在：

- 官腔
- 训话
- 阴阳怪气

这样短时间很有趣，但很快会腻。

真正耐玩的版本，应该同时有三种东西：

- `压迫感`：你知道这事可能升级
- `人情味`：有些辅导员确实是在帮你收场
- `规则感`：不是看心情，而是能感到制度在推动剧情

一旦这三者同时存在，用户会觉得这个世界“像真的”。

## 14. 任务规划：按阶段落地

如果我们下一步真要开做，我建议分 5 个阶段，不要一开始把所有系统一起上。

### Phase 1: 把产品骨架搭起来

目标：

- 明确 skill 名称和定位文案
- 写出基础 `SKILL.md`
- 定 4 个核心辅导员 archetype
- 写 6 个经典场景
- 跑通单场景对话闭环

交付物：

- `SKILL.md`
- `references/counselor-archetypes.md`
- `references/scene-library.md`
- 基础 scene prompt

### Phase 2: 先把“自定义人格”做成主入口

目标：

- 完成快速创建问卷
- 完成高级人格维度配置
- 把 archetype + 滑块 + 标签合成一个 persona
- 加 3 个测试预览场景

交付物：

- `prompts/custom_intake.md`
- `prompts/archetype_picker.md`
- `prompts/preview_runner.md`
- `generated/personas/` 的输出结构

这是最关键的版本节点。
做到这里，项目就已经成立了。

### Phase 3: 再接入蒸馏增强

目标：

- 支持导入通知、聊天、口述材料
- 提取真实说话风格和规则倾向
- 合并到已有 persona
- 支持用户纠偏

交付物：

- `tools/notice_parser.py`
- `tools/chat_parser.py`
- `prompts/persona_calibrator.md`
- `prompts/distillation_merger.md`

### Phase 4: 提升可玩性和沉浸感

目标：

- 加结果分级
- 加案件档案
- 加复盘模式
- 加多角色链路
- 加长期印象分系统

- `prompts/outcome_judge.md`
- `prompts/debrief.md`

### Phase 5: 做进化和扩展能力

## 15. 建议的执行顺序

如果按性价比排序，最应该优先做的是：

1. 名字和定位
2. 4 个 archetype
3. 6 个场景
4. 自定义人格创建流程
5. 预览和纠偏
6. 蒸馏增强
7. 结果分级和复盘
8. 多角色场景

原因很简单：

- 没有 archetype，就没有世界观
- 没有场景，就没有玩法
- 没有自定义人格，这个项目的关键交互就立不住
- 没有预览和纠偏，用户会觉得“不像”
- 蒸馏很重要，但应该建立在一个已经能玩的核心产品之上
## 16. 一句话总结

最优解不是做一个“会说官话的辅导员”，而是做一个：

> 有人设、有规则、有证据链、有分支结果、还能复盘的高校生存模拟 Skill。

这才会真的有趣、好玩、耐玩，也更容易做成一个能持续扩展的系列。
