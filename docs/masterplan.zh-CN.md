# 总规划

这份文档是整个项目的总规划。

它要回答 3 个问题：

1. 这个项目最终想成为什么
2. 长期应该先做什么、后做什么
3. 当后续只收到一句“继续”时，默认该沿哪条主线推进

## 1. 最终目标

最终版本不应只是：

- 一个会演辅导员的 prompt
- 一个困难沟通排练器
- 一个说明文生成器

而应是：

> 一个可为用户生成、维护、蒸馏、纠偏、记忆并持续进化专属辅导员交互系统的 meta-skill。

## 2. 最终版本长什么样

最终版本应至少具备 6 层：

### 层 1：专属人格层

- archetype
- 手工设定
- 蒸馏增强
- correction

### 层 2：系统模式层

- 开口器
- 追问应对器
- 办公室推演器
- 补救器
- 说明文工坊
- 求助沟通器
- 争取通融器
- 复盘器
- 会话记忆器
- 人格进化器

### 层 3：规则宇宙层

- 学校规则
- 学院气质
- 风险升级路径

### 层 4：案件状态层

- 已知信息
- 隐藏证据
- 风险等级
- 结果档位

### 层 5：长期记忆层

- sessions
- corrections
- 用户沟通偏好
- 历史高频问题

### 层 6：发布与展示层

- GitHub 首页
- 示例系统
- 示例链路
- 示例输入输出

## 3. 长期开发顺序

### 第一优先

- 用户到底怎么开始
- 第一次能不能立刻拿到可用内容

### 第二优先

- 同一事件在不同模式下是否真的不同
- 系统模式是否比固定模板更有价值

### 第三优先

- correction 和 session memory 是否真的能积累
- 用户第二次再来时，系统是否已经变了

### 第四优先

- 深场景闭环
- 单辅导员一致性强化
- 学校差异模板

## 4. 默认推进规则

如果后续只收到一句“继续”，默认按这个优先级推进：

1. 优先补最短板的系统闭环
2. 优先补最值钱的系统级示例
3. 优先补最影响首次体验的内容
4. 再补深场景和复杂工具
5. 最后补外围文档和装饰层

## 5. 当前推荐下一步

从当前仓库状态出发，最值得做的不是再加一堆事件，而是：

1. 做更多“同一事件在不同模式下”的对比样例
2. 做一个真的会读写 session 的系统演示
3. 让一个辅导员系统经历多轮使用后明显变化
4. 再往单辅导员一致性和长期档案推进

## 6. 现在的默认长期推进法

如果后续只收到一句“继续”，默认按下面这条主线推进：

1. 先补系统中枢和生命周期，让项目更像一个系统
2. 再补能直接展示系统价值的样例和快照
3. 再补真正影响连续使用体验的记忆、纠偏、蒸馏闭环
4. 再补深场景、单辅导员一致性、长期档案
5. 最后补学校差异模板和发布强化层

更细的 8 轮推进线见：

- [final-version-track.zh-CN.md](./final-version-track.zh-CN.md)

## 7. 最终最好版本的判断标准

最终最好版本不是“文件更多”，而是同时做到下面几件事：

- 新用户第一次用就能拿到可用内容
- 老用户第二次回来时能感到系统延续
- correction 会让系统真的改变
- 蒸馏会让系统更像某个学校生态
- 深场景会维护状态、证据、风险和结果
- 仓库示例会让陌生人一眼看懂这套系统的价值
- 不依赖多角色扩展，也能靠单一辅导员系统撑起完整体验
