# 发布前我自己怎么用

这份文档是写给仓库作者自己的。

你在发布之前，不需要先把所有工具链都跑一遍。
更合理的方式是，先按真实用户路径自己用 3 到 4 次，确认这套系统真的顺。

## 1. 最省事的自测方式

在当前工作区里，直接对 Codex 或 Claude Code 说：

```text
请使用当前仓库里的 skill，先帮我快速创建一个基础辅导员系统。
我的导员属于那种平时不怎么发火，但会一步一步把问题压实的人。
我现在卡在：已经给他发过请假消息了，但他回得很冷，我不知道下一句怎么接。
```

这一步的目的不是追求一开始就“特别像”，而是先确认：

- 入口是不是顺
- 第一轮结果是不是能直接用
- 系统有没有自然进入 `follow-up` 之类的模式

## 2. 第二次自测：补一点真实素材去蒸馏

然后再补一点你自己手头真的有的素材，比如：

- 一段私聊
- 一条通知
- 你对这个导员的手打印象

直接说：

```text
我现在补三类素材给你，请你基于当前仓库里的蒸馏流程，更新这个辅导员系统。
重点校准三件事：
1. 他第一句话怎么开口
2. 他追问时怎么给压力
3. 他最后怎么收口
更新后告诉我：哪些地方已经适合现实同步，哪些地方还不够真。
```

这一步主要看：

- 蒸馏有没有真的改系统
- 输出会不会只改措辞，不改处理逻辑
- 真实性反馈是不是足够诚实

## 3. 第三次自测：拿一个现实问题去练

选一个你真的可能会遇到的沟通问题，不一定是请假。

比如：

- 请假
- 被追问缺勤
- 说明文
- 求助
- 想争取一次通融

直接说：

```text
现在不要泛泛分析，直接用这个辅导员系统陪我练一轮真实沟通。
目标不是好看，而是给我一条今天就能发的消息，一句被追问后的下一句，以及一个最容易翻车的提醒。
```

如果这一轮出来以后，你觉得“确实更接近能拿去现实里用了”，说明主线成立。

## 4. 第四次自测：把现实反馈带回来

等你真的用了以后，把真实回复贴回来：

```text
这是现实里辅导员刚刚回我的原话。
请不要重新从头分析，请按当前仓库的回流逻辑处理：
1. 写这一轮 session summary
2. 如果哪里不像，写 correction
3. 如果这件事已经形成长期痕迹，更新 archive
4. 最后告诉我下次更适合从哪个模式进入
```

如果这一轮走顺了，这个仓库就不再只是“演练器”，而是真的具备了长期系统感。

## 5. 发布前最值得你亲自验证的 5 件事

1. 第一轮是不是不用长访谈，也能给出可用结果。
2. 蒸馏后是不是真的更像，而不是只是更花。
3. 现实同步输出是不是能直接复制或轻改后使用。
4. 真实反馈回来之后，系统是不是真的变了。
5. 开口、追问、说明文、复盘，是不是还是同一个导员。

## 6. 建议你实际就按这 4 轮用

最推荐的发布前自测顺序：

1. `从零创建`
2. `补素材蒸馏`
3. `拿真实问题演练`
4. `带真实反馈回流`

如果这四轮你自己都愿意连续用下去，再发 GitHub 会更稳。
