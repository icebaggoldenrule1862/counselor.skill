# 长期任务路线图

这份路线图用于固定长期推进顺序。

## Release 0.1: Demo 基础版

目标：

- GitHub 首页完整
- Skill 入口完整
- archetype、场景库、示例人格、示例对局齐备
- 第一版蒸馏工具可跑

状态：

- 已完成

## Release 0.2: 沟通起手版

目标：

- 先把“用户怎么开始”做清楚
- 从沟通阶段而不是事件名进入
- 让第一次使用就能拿到第一份可用输出

关键任务：

- 定义起手路由
- 定义高价值排练模式
- 写首批第一句排练、追问应对、约谈预演、补救沟通样例
- 把 README 和 SKILL 改成按沟通节点组织

状态：

- 已完成第一轮修正

## Release 0.3: Meta-skill 定位版

目标：

- 从“排练器”抬到“专属辅导员系统生成器”
- 补系统模式层
- 补 session summary 与会话记忆脚手架
- 补系统生命周期和中枢调度层

关键任务：

- 定义 system modes
- 定义 system lifecycle
- 定义 orchestrator
- 加入 session summary prompt
- 加入 session writer 工具
- 把 README、SKILL、vision 统一到 meta-skill 定位

状态：

- 已完成

## Release 0.4: 系统闭环演示版

目标：

- 让系统“连续使用会变化”这件事被看见
- 补齐 snapshot、系统演化链路和更完整的多轮样例
- 让 GitHub 访客一眼能看出这不是 prompt 拼盘

关键任务：

- 增加完整系统快照工具
- 增加完整系统样例目录
- 做同题不同模式对比样例
- 做连续 2 到 3 次使用后的系统变化演示

## Release 0.5: 深场景版

目标：

- 形成稳定的案件状态文件结构
- 至少 3 个深场景有完整状态机
- 结果档位、风险变量和复盘串起来

关键任务：

- 规范 `generated/cases/` 结构
- 固化状态字段
- 为高压力场景写完整流程
- 把案例状态、session 和记忆连接起来

## Release 0.6: 人格进化版

目标：

- correction 机制更像真正的进化层
- persona 可以持续演化并反映到不同模式

关键任务：

- 强化 correction 优先级
- 做“修正前 / 修正后”对比链路
- 让 correction 影响开口、追问、预演、复盘四类输出

## Release 0.7: 蒸馏增强版

目标：

- 扩大蒸馏输入类型和质量
- 让导入材料前后人格和规则差异明显

关键任务：

- 通知、聊天、制度文本的合并逻辑
- 冲突提示
- 一套完整蒸馏演示链路

## Release 0.8: 会话记忆版

目标：

- 用户不是每次都重新开始
- session summary 能沉淀出长期训练轨迹
- 后续建议能自然续接上一次

关键任务：

- 定义 sessions 目录结构
- 定义 summary 的读写规则
- 让下一轮排练可自然续接上一次
- 把快照和 session 统计接到展示层

## Release 0.9: 单辅导员一致性强化版

目标：

- 让同一个辅导员在不同模式和不同场景下都保持高度一致
- 不依赖扩角色，也能提升耐玩度和真实感

关键任务：

- 做开口 / 追问 / 约谈 / 说明文 / 复盘跨模式一致性检查
- 做同一导员跨模式前后对比样例
- 把 correction、蒸馏、长期档案都映射进这套一致性验证

## Release 1.0: 长期档案版

目标：

- 用户不是“每局新号”
- 新场景能继承过去记录
- 系统能体现长期印象和风险标签变化

关键任务：

- 定义学生长期档案结构
- 定义印象分和风险标签变化
- 把过去记录带入新场景

说明：

- 当前已经开始提前铺设这层基础，包括 archive schema、archive writer 和样例 archive

## Release 1.1: 学校差异模板版

目标：

- 让不同学校和学院气质开始分化

关键任务：

- 抽象学校差异参数
- 做 3 套以上学校氛围模板
- 提供模板示例

## Release 1.2: GitHub 发布强化版

目标：

- 让陌生访客在首页 1 分钟内理解项目
- 让仓库样例、快照、系统演化、模式对比彼此串起来

关键任务：

- 优化首页信息密度
- 增加更完整的系统样例和截图式演示
- 整理首版发布清单和展示顺序

## 持续质量任务

- 精炼 README 和示例
- 保持目录整洁
- 验证 Windows 编码兼容
- 增加脚本测试样例
- 保持示例输出可复现
- 持续避免功能扩散

## 默认推进顺序

如果后续只收到“继续”，默认按这个优先级推进：

1. 先补最短板的系统闭环
2. 再补最值钱的示例
3. 再补最影响体验的工具
4. 再补长期记忆和版本化能力
5. 最后再补外围文档
