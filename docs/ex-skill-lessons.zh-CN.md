# 从 ex-skill 学到的架构方法

这份文档专门回答：

> `ex-skill` 为什么题材很强、体验很有趣，但并没有失控扩展成一堆散功能？

我重新读了一轮它的 `SKILL.md`、`PRD`、`memory/persona analyzer`、`merger`、`scene_director` 之后，结论很清楚：

它真正强的不是“前任”这个题材，而是它始终守住了一个非常稳的系统骨架。

## 1. 核心系统非常少，但很稳

`ex-skill` 的核心其实只抓了几件事：

1. 基础 intake
2. 原材料导入
3. 分成 `Memory` 和 `Persona` 两条线分析
4. merge
5. correction
6. version / session evolution

也就是说，它不是看到一个玩法就立刻加一个顶层模块。
它先保证：

- 同一个角色能被创建
- 能被修正
- 能被追加材料
- 能被持续使用

这就是它稳定的原因。

## 2. 它把“运行核心”和“玩法模式”分开了

最值得学的一点是：

`scene_director` 虽然有趣，但它在 `ex-skill` 里并不是顶层主干，而是一个可选场景模式。

这很关键。

因为它说明：

- 有趣玩法可以有
- 但不能反过来绑架核心架构

翻到我们这里，现在更明确的结论是：

- 我们不继续做多角色主线
- 我们真正要学的是“玩法不能绑架主干”
- 所以后续全部收口到“单一辅导员系统的一致性和进化”

## 3. 它的“有趣”来自一致性，不来自到处扩展

`ex-skill` 好玩的地方，不是功能菜单多。

而是：

- 这个人设像不像
- 追加材料后会不会更像
- correction 后是不是马上变了
- scene 模式里还能不能保持同一个人味

也就是说，它的趣味来自：

> 同一个系统在不同情境里都保持一致性。

这对我们特别重要。

我们这个项目以后真正的趣味，也不该主要来自：

- 事件越来越多
- 角色越来越多
- 菜单越来越长

而应该来自：

- 同一个导员在开口、追问、约谈、说明文、复盘里都很像
- correction 后全系统都跟着变
- 蒸馏后整个学校生态更真实

## 4. 多源输入不是为了“什么都支持”，而是为了分层提取

`ex-skill` 的 parser 不是乱堆的。

它每种来源都在回答不同问题：

- 聊天记录：说话方式
- 照片：时间线
- 社交内容：公开人设
- 用户口述：主观记忆补洞

这也是我们现在扩蒸馏时最该坚持的原则：

- 通知不是拿来证明“这个人到底凶不凶”的唯一来源
- 私聊不是拿来代表全部规则层
- 手打印象不是拿来替代全部证据

而应该是：

- 通知提公开语气
- 私聊提追问方式
- 制度文本提规则层
- 班会/约谈转写提压力节奏
- 批注提说明文标准
- 用户手打印象提高优先级行为约束

## 5. 它的 merge 很克制

`ex-skill` 的 `merger` 一个很重要的态度是：

- 增量不覆盖一切
- 冲突要标记
- 新证据是补强，不是乱洗

这也提醒我们：

以后无论是蒸馏、深场景还是新场景追加，
都要尽量做成：

- 在系统主干上补层
- 而不是把原有结构推翻重写

## 6. 对我们项目最重要的启发

结合上面这些，我对我们这个项目的修正判断是：

### 判断 A：放弃多角色，把所有资源集中到单一辅导员系统

这不是退一步，而是更清楚地抓主干。

因为这个项目最强的体验，从头到尾都不是“角色数量”，而是：

- 这个导员像不像
- correction 后是不是全系统都变了
- 蒸馏后是不是更像学校生态
- 连续几次使用后是不是已经有历史感

## 7. 我们后续该怎么改

所以后续真正该做的，是把 `Round 6` 改成“单辅导员一致性强化”：

1. 让开口、追问、约谈、说明文、复盘都像同一个人
2. 让 correction 稳定改写多种模式
3. 让蒸馏进入规则层、口气层、施压节奏层
4. 让长期档案真正改变下一次进入时的系统状态

## 最终结论

`ex-skill` 真正该学的不是题材，也不是表面功能。

而是这句话：

> 核心系统要小而稳，一切扩展都必须服从主干。

这会直接决定我们这个项目后面是越做越强，还是越做越散。
