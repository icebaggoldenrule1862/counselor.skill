# 发布检查清单

这份清单只服务一件事：

> 在真正发到 GitHub 之前，确认这个仓库已经像一个成品，而不是一堆内部资料。

## 1. 首页检查

- [ ] `README.md` 开头三屏内能回答“这是什么”
- [ ] `README.md` 明确写出“不是模板库，而是单一辅导员系统”
- [ ] `README.md` 有清楚的起手入口
- [ ] `README.md` 有蒸馏真实性入口
- [ ] `README.md` 有现实同步入口
- [ ] `README.md` 有完整系统样例入口
- [ ] `README.md` 有边界说明

## 2. 首次使用检查

- [ ] 新用户能通过 [examples/onboarding/README.md](./../examples/onboarding/README.md) 走完首次使用链路
- [ ] 新用户能看懂先补哪些素材
- [ ] 新用户能理解“先蒸馏，再验真，再现实同步”的顺序
- [ ] 新用户不会被迫先理解全部目录结构

## 3. 蒸馏检查

- [ ] 至少有一套清楚的蒸馏操作说明
- [ ] 至少有一份真实性协议
- [ ] 至少有一份真实性报告样例
- [ ] README 能解释为什么“像”不只是口头禅像

## 4. 系统样例检查

- [ ] `quiet-pressure-system` 可以被当成展示样板阅读
- [ ] 样例里同时包含 persona、archive、sessions、corrections、snapshot
- [ ] 样例能展示蒸馏前后变化
- [ ] 样例能展示 correction 前后变化
- [ ] 样例能展示真实反馈回流

## 5. 现实同步检查

- [ ] 不只展示请假
- [ ] 至少覆盖 follow-up、help、statement、debrief 中的多个用途
- [ ] 读者能理解“模拟结果如何拿去现实里用”
- [ ] 读者能理解“现实里用了以后如何带回来”

## 6. 一致性检查

- [ ] 模式虽然不同，但仍然像同一个辅导员
- [ ] correction 是跨模式生效的
- [ ] 文档里没有残留“多角色联动”主线
- [ ] 顶层叙事始终是单一辅导员系统

## 7. 文档收口检查

- [ ] 有最终目标文档
- [ ] 有发布地图
- [ ] 有能力审视文档
- [ ] 有 `v1.0` 验收页
- [ ] 有这份发布检查清单
- [ ] 有发布前自测说明

## 8. 最后判断

如果上面大多数项都已经打勾，可以认为仓库已经接近：

> 一个适合放到 GitHub 首页展示、也能让陌生用户实际跟着走起来的 `v1.0`。
